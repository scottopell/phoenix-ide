# Auto-stash uncommitted worktree changes on ContextExhausted cleanup

## Problem

Task 24692 shipped worktree cleanup on the `ContextExhausted` transition
for Work/Branch mode conversations. Cleanup runs inside
`Effect::NotifyContextExhausted` at `src/runtime/executor.rs:1155` and
calls `git_ops::remove_worktree_preserve_branch` via `--force`. Any
uncommitted changes in the worktree are destroyed.

The context-full banner renders **after** the cleanup (the
`StateChange` SSE fires after the blocking git op returns), so the user
gets no opportunity to commit or stash externally. Agents commit
frequently per repo convention, but nothing enforces it -- an LLM that
just finished a big patch sequence without committing is exactly the
scenario most likely to also blow past the context window.

## Decision (from conversation on 2026-04-22)

Auto-stash before the `git worktree remove`. Preserves the work in
git's stash stack; user recovers via `git stash list` / `git stash
apply <sha>`. Preferable to deferring cleanup (which reintroduces the
orphan-worktree state we just eliminated) and to warning-only (which
does nothing to actually preserve the data).

Second decision: the stash ref lands in the continuation summary (the
text that becomes the seed-draft of the "Continue in new conversation"
flow). When the user clicks Continue, the new agent's first message
contains the stash ref and instructions, so the continuation can begin
with `git stash apply <sha>` and resume exactly where the previous
agent left off -- no manual user step.

## Implementation

### Backend

1. Before `remove_worktree_preserve_branch` runs in
   `Executor::cleanup_context_exhausted_worktree`, detect uncommitted
   changes via `git status --porcelain --untracked-files=all` in the
   worktree. Empty output = nothing to stash, proceed as today.

2. If there are changes:
   ```
   git -C <worktree_path> stash push --include-untracked \
       --message "phoenix context-exhausted <conv_id>"
   ```
   Then capture the stash's commit SHA with `git rev-parse stash@{0}`.
   The SHA is durable (resolvable even after other stashes are pushed);
   the `stash@{N}` label is not.

3. On stash failure, log at warn level and continue the cleanup. The
   cleanup is best-effort throughout -- losing data silently is worse
   than losing it with a warning, but blocking the terminal transition
   would be worse still.

4. Persist the stash SHA somewhere the continuation flow can reach.
   Recommended: extend the `ContextExhausted` state payload.
   Currently `{ summary: String }`; becomes
   `{ summary: String, stashed: Option<StashRef> }` where
   `StashRef = { sha: String, worktree_path: String }`. Keeps state
   self-describing; aligns with the correct-by-construction principle
   (no parallel representations, no serde-default shim). Alternative
   (new DB column on conversations) is possible but the state payload
   already persists and is already serialized for SSE and restart
   restoration, so reusing it is cleaner.

5. Thread the stash ref into the continuation summary. The summary is
   generated by the LLM via the continuation-request flow
   (`src/state_machine/transition.rs:1511` and nearby). Append a
   structured footer after the LLM-generated text:
   ```
   ---
   Prior worktree had uncommitted changes at context exhaustion.
   They were stashed to preserve them:

     stash SHA: <sha>
     prior worktree path: <path>

   To resume from exactly where the previous agent left off, apply the
   stash in the new worktree:

     git stash apply <sha>

   Resolve any conflicts before continuing.
   ```
   Append happens server-side where the final summary is assembled, so
   the FE's existing seed-draft hydration picks it up for free.

### Frontend

1. When `ContextExhausted.stashed` is present in the state payload,
   render a small line under the banner header: "Uncommitted changes
   stashed: `<sha[:7]>` -- recover with `git stash apply <sha>`."
   Informational only.

2. No change to the Continue button -- the stash ref is already in the
   seed-draft, which auto-populates the new conversation's input area.

### Tests

- Unit: `cleanup_context_exhausted_worktree` with a dirty worktree
  produces a stash entry; branch still exists; worktree is gone.
- Unit: with a clean worktree, no stash is created (noise avoidance).
- Integration: full continuation flow -- parent becomes
  `ContextExhausted`, banner shows stash ref, new conv's seed-draft
  contains `git stash apply <sha>`.
- Spec update: add `ensures: if worktree_dirty then Stashed(sha)` clause
  to `WorkBranchWorktreeCleanupOnContextExhausted` in
  `specs/projects/projects.allium`.

## Out of scope

- Auto-applying the stash on the continuation side. The new conv runs
  in a fresh Branch-mode worktree; applying the stash there is the
  user's (or the agent's) choice. The summary footer tells the agent
  what to do -- let it decide.
- Stash pruning / GC. Stashes accumulate in `git stash list`. Task
  08632 (done) is adjacent but separate; file a follow-up if stash
  hygiene becomes a problem.
